## 它到底解决什么问题

网页抓取最烦的不是写解析逻辑，而是处理反爬。验证码、IP 封禁、浏览器指纹检测、JavaScript 渲染——每一项都能让你的爬虫在凌晨三点挂掉。ScraperAPI 把这些全部打包成一个 API 调用：你发一个 GET 请求，它返回渲染完成的 HTML。

我自己的使用场景很直接：

1. 把目标 URL 拼进 ScraperAPI 的端点
2. 加上参数指定是否需要 JavaScript 渲染、地理位置定向
3. 拿到干净的 HTML，交给解析器处理

整个过程不需要维护代理池，不需要管 Puppeteer 的内存泄漏，不需要半夜爬起来重启服务。

## 核心能力拆解

### 智能代理轮换与反封禁

ScraperAPI 在后台维护着超过 4000 万个 IP 地址的代理池，覆盖住宅 IP、数据中心 IP 和移动 IP 三种类型。每次请求自动轮换，失败自动重试。我跑了 30 天，成功率稳定在 99.2% 左右——这个数字是我在 dashboard 里看到的实际统计，不是宣传页上的。

### JavaScript 渲染

很多现代网站的内容是客户端渲染的，普通 HTTP 请求拿到的是空壳。ScraperAPI 支持 `render=true` 参数，后台用无头浏览器帮你执行 JS 后再返回完整 DOM。代价是每次调用消耗的 API 额度会翻倍（算 10 个 credit），但省下来的 Puppeteer 运维成本远超这点消耗。

### 地理位置定向

需要抓取特定国家的本地化内容？加一个 `country_code=us` 参数就行。支持超过 50 个国家和地区的 IP 出口。我做东南亚电商数据时用过 `country_code=sg`，拿到的价格和新加坡用户看到的完全一致。

### 结构化数据端点

除了通用的 HTML 抓取，ScraperAPI 还提供针对亚马逊、Google 搜索结果的结构化 JSON 端点。直接返回解析好的产品标题、价格、评分，省掉你写 XPath 的时间。

## 我踩过的坑和你可以绕开的弯路

### 坑一：JavaScript 渲染的 credit 消耗

开了 `render=true` 之后，每次请求消耗 10 个 credit 而不是 1 个。我第一周没注意这个细节，月中就用掉了 60% 的额度。后来我的做法是：先用普通请求试一次，如果返回的 HTML 里目标数据为空，再回退到 JS 渲染模式。这样能省下大约 40% 的 credit。

### 坑二：并发数不是越高越好

Startup 计划给了 50 并发，但我一开始就把线程池拉满，结果目标网站的响应时间飙升，成功率反而下降。后来我把并发控制在 20-30 之间，成功率回到 99% 以上。ScraperAPI 的重试机制会自动处理失败请求，但如果你的并发太激进，重试队列会堆积，整体吞吐量反而下降。

### 坑三：地理定向的额外消耗

指定 `country_code` 会额外消耗 credit（住宅 IP 定向消耗更多）。如果你的场景不需要精确到国家级别的 IP 出口，别加这个参数，让 ScraperAPI 自动选择最优路由。
